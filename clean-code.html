01 · Principle

Readability First

Your code will be read far more times than it’s written. Every minute you spend making it easier to read pays back many times over in the reads, reviews, and changes that follow.

Key Insight

“I can figure it out if I read it carefully enough” is the voice of someone paying interest on bad code. A senior engineer writes code that a tired teammate can read on a Friday afternoon.

What readable code actually looks like

  • Short functions — one responsibility, ideally fits on a screen.
  • Clear names — the reader shouldn’t have to look up what a variable or function means.
  • Boring structure — the least clever solution that works. Surprising code is a tax on every future reader.
  • Local reasoning — a function does what its name says, without hidden side effects far away.
  • Obvious flow — the happy path reads top to bottom; errors are handled clearly; nothing important hides in a callback five levels deep.
📖

Test for yourself: read your own code tomorrow morning before your first coffee. If you stumble on a line, that line will confuse your teammate too. Fix it before you submit.

02 · Craft

Naming as a Design Decision

A good name removes the need for a comment. A bad name forces every future reader to rebuild your intent from scratch. Naming is not cosmetic — it is design.

Name patterns worth practicing

Domain over implementation

cancelSubscription() beats updateStatus(true). The first tells the reader what the business is doing. The second only tells them what the code is doing.

Booleans that read as questions

isEligible, hasExpired, canRefund. Not active or flag. The reader should know from the name what true means.

Verbs for actions

sendInvoice, fetchUser, retryWithBackoff. A function is a verb; a variable is a noun. Keep the parts of speech straight and the code reads like prose.

No filler words

UserManager, OrderHelper, DataProcessor are names that mean nothing. If you can’t say what it does without hedging, you haven’t found its real responsibility yet.

Name smells

data, info, obj, temp, stuff. Single-letter names outside a very tight loop. Names that require the reader to remember context from three files away.

Better names

Names that answer “what is this and why is it here?” on their own. Names that survive a code reviewer reading only the patch, without the surrounding file.

03 · Principles

Principles Worth Holding

These are the short principles senior engineers quote to each other. Each one is a lens, not a law. Used as a lens they make you sharper; used as a law they become cargo cult. Know the why behind each.

DRY — Don’t Repeat Yourself

What it means: every piece of knowledge should have a single, unambiguous home. Duplicated knowledge drifts out of sync; duplicated structure is often fine.

Why: when the same business rule lives in three places, every change to that rule requires finding and editing three places — and missing one is the bug.

The misuse: extracting a helper the first time two lines look alike. Two methods that happen to do similar things today but will diverge tomorrow are better left duplicated. Sandi Metz: “Duplication is far cheaper than the wrong abstraction.” Wait for the third duplicate before abstracting.

KISS — Keep It Simple

What it means: prefer the simplest thing that could possibly work. Complexity has to earn its place.

Why: simple code is easier to read, change, debug, and test. Every piece of complexity you add is a small tax on every future reader and every future change.

The misuse: “simple” becoming an excuse for under-engineering. A simple solution that silently corrupts data is not simple — it’s cheap. Simple means as simple as the problem allows, not simpler.

YAGNI — You Aren’t Gonna Need It

What it means: don’t build for hypothetical future requirements. Build for the ones that actually exist today.

Why: most “we might need this later” extension points are never used. They add complexity now in exchange for flexibility you never spend. When the real need arrives, you’ll design it better with real information — and the old speculation will already be in the way.

The misuse: using YAGNI to dodge obvious near-term needs. If you know a second type of payment provider is arriving next sprint, building the one-provider interface as if it were permanent is not YAGNI — it’s wilful ignorance. YAGNI is about resisting speculation, not about resisting foresight.

Tell, Don’t Ask

What it means: tell objects what to do; don’t ask them for their state and make decisions on their behalf. The behavior belongs with the data that supports it.

Why: when you “ask and decide”, the same decision tends to happen in many callers — and drift. When you “tell”, the rule lives inside the object, enforced everywhere the object is used.

Ask — knowledge leaks out 
if (order.getStatus() == PAID
    && !order.isShipped()
    && order.getPayment() != null) {
  shippingService.ship(order);
}
Tell — knowledge lives with order 
if (order.isReadyToShip()) {
  shippingService.ship(order);
}
// Order owns the rule;
// callers don't re-derive it.

Law of Demeter — Don’t Talk to Strangers

What it means: an object should only call methods on: itself, its parameters, objects it creates, or its direct fields. Not on things it reached through another object.

Why: order.getCustomer().getAddress().getCity() couples the caller to the entire chain. A change to Customer’s internals ripples to everyone who reached through it. Expose order.getShippingCity() instead.

The misuse: treating it as a blanket ban on chained calls. Fluent APIs (stream().filter().map()) chain because each call returns the same abstraction; that’s not a Demeter violation. The law is about reaching through different objects, not about method chains per se.

Composition over inheritance

What it means: default to building behavior by assembling smaller objects, not by extending a base class.

Why: inheritance is a very strong coupling — it commits you to the parent’s interface, state, and invariants forever. Composition lets you mix-and-match capabilities and change them without rewriting a type hierarchy. Most long-lived codebases regret deep inheritance hierarchies; few regret composition.

💡

These principles contradict each other in any specific code review. DRY says “extract this!” and YAGNI says “don’t!” — both reasonable. Judgment is knowing which one applies here, and being honest when the answer is “I don’t know yet; let’s defer.”

04 · Principles

SOLID in Practice

SOLID is often taught as five rules to memorise. Better to hold it as five questions you ask while designing a module.

S — Single Responsibility: does this class have one reason to change? ▾

If two different kinds of business change both force edits to the same class, the class is doing two jobs. Split them. The test isn’t “does it do one thing?” — it’s “does it have one reason someone would change it?”

O — Open/Closed: can I extend this without editing it? ▾

If every new kind of thing requires adding a new if branch in the same switch statement, the design resists extension. Use polymorphism or a strategy pattern so the core stays stable and new cases live in their own classes.

L — Liskov: can a subclass stand in for its parent without surprises? ▾

A subclass that throws where its parent doesn’t, or rejects arguments the parent accepts, breaks the contract. If your subtype is fundamentally different, don’t inherit — compose or use a separate interface.

I — Interface Segregation: is this interface asking callers to depend on things they don’t use? ▾

A 20-method interface where each caller uses three of them creates false coupling. Prefer several small, focused interfaces over one god-interface.

D — Dependency Inversion: does high-level code depend on abstractions or concrete details? ▾

Business logic should depend on abstractions (a PaymentGateway interface), not on a specific library (StripeSDK). This isn’t about testability alone — it’s about keeping the domain isolated from churn in the adapter layer.

⚠️

SOLID is a prompt, not a cage. Over-abstracting to satisfy a principle creates more complexity than the smell it was fixing. Apply the principles when the design starts to hurt — not pre-emptively on every three-line helper.

05 · Modelling

Domain Modelling — Rich Models, DDD Lite

The most important thing most backends do is represent a domain: orders, users, payments, bookings. How faithfully your code mirrors that domain is how easy (or impossible) it will be to change the business rules three years from now. Senior engineers treat the domain model as a first-class artifact — not as a thin layer over the database.

Key Insight

Domain classes that only hold data — no behavior, just getters and setters — look “clean” but are the most common anti-pattern in enterprise code. The business rules end up scattered across services, repeated, drifting. Put behavior where the data lives.

The Anemic Domain Model — the trap to avoid

What it looks like: Order has getStatus(), setStatus(), getItems(), setItems(), and no methods that do anything. The real logic — “can this order be shipped?”, “what’s the total?”, “apply this discount” — lives in OrderService or scattered across layers.

Why it’s a trap: the object loses its meaning. It’s no longer an Order; it’s a bag of fields the services fight over. Every caller has to know which service to go to for which behavior, and those services end up duplicating checks that should live once on the object.

What Martin Fowler calls it: “contrary to the basic idea of object-oriented design, which is to combine data and process together.” Read his AnemicDomainModel bliki entry when you have 10 minutes.

Anemic 
class Order {
  List<Item> items;
  Status status;
  // only getters/setters
}

class OrderService {
  boolean canShip(Order o) {
    return o.getStatus() == PAID
      && !o.getItems().isEmpty()
      && o.getPayment() != null;
  }
  Money totalOf(Order o) { /* ... */ }
}
Rich — behavior on the data 
class Order {
  List<Item> items;
  Status status;

  boolean canShip() {
    return status == PAID
      && !items.isEmpty()
      && payment != null;
  }
  Money total() { /* ... */ }
}
// OrderService orchestrates;
// Order knows itself.

DDD Lite — the useful parts of Domain-Driven Design

You don’t need to adopt the full Evans blue book to benefit from DDD. A few ideas land almost every feature:

Four DDD ideas worth carrying

  • Ubiquitous language. Use the business’s words in the code. If PMs say “subscription”, the class is called Subscription, not RecurringBillingItem. Translation layers between code and conversation are where misunderstanding breeds.
  • Entities vs. value objects. An entity has identity that persists (User, Order). A value object is defined by its contents (Money, Address, DateRange). Value objects should be immutable; it simplifies almost everything downstream.
  • Aggregates. A cluster of objects treated as one for change purposes. You modify the aggregate through its root (Order), which enforces invariants. Prevents “someone reached in and broke a rule” bugs.
  • Bounded contexts. The same word means different things in different parts of the business. “Customer” in sales is not the same as “Customer” in billing. Don’t force one shared model; let each bounded context own its version.
📊

Scenario: you’re modelling a booking system. The anemic move: Booking is a dataclass; BookingService has cancel(booking), extend(booking, hours), isActive(booking). The rich move: Booking itself knows how to cancel(), extend(hours), answer isActive() — and enforce “can’t extend a cancelled booking” in the method itself. The rich model makes bad calls impossible; the anemic one makes you wish you’d caught them in review.

Common pitfall

Full DDD kit (aggregates, repositories, domain events, CQRS, event sourcing) applied to a CRUD app. The ceremony overwhelms the domain; the team spends more time on infrastructure than on the business.

Better approach

Start with rich domain objects and ubiquitous language — the two highest-value ideas. Reach for aggregates when invariants span multiple objects. Reach for bounded contexts when your “Customer” means five different things in five places. Don’t adopt the full method unless the problem needs it.

📖

Deeper reading: Evans’s Domain-Driven Design is the canonical text but dense. Vernon’s Implementing Domain-Driven Design is more practical. Fowler’s Patterns of Enterprise Application Architecture covers Anemic vs. Rich models and the standard domain-layer patterns. See resources.

06 · Habit

Refactoring as a Habit

Refactoring is not a quarterly project. It’s a daily habit: each change leaves the area slightly better than you found it. The compound is what keeps a codebase healthy for years.

The Boy Scout rule, applied

  • Rename a misleading variable while you’re passing through.
  • Extract a helper when a function grows past the screen.
  • Delete the commented-out block you just confirmed isn’t needed.
  • Replace a magic number with a named constant.
  • Move a misplaced method to the class that actually owns it.

None of these should bloat your PR. Keep them tight, keep them scoped, and call them out in the description so reviewers see them intentionally.

Refactor first, then change

When a change is hard, the honest move is often to refactor the surrounding code into a shape where the change becomes easy, then make the easy change. Kent Beck’s version: “Make the change easy, then make the easy change.”

Common pitfall

Mixing a big refactor and a feature change in the same PR. Reviewers can’t separate what’s behavior change from what’s moving chairs around. Both halves get rubber-stamped or both get rejected.

Better approach

Land the refactor as its own PR (ideally behavior-preserving, with tests green before and after). Then land the feature on top. Each PR is small, each is reviewable.

🔧

Refactor-to-understand: when you’re struggling to follow a piece of code, rewrite it for yourself. Don’t commit the rewrite; the goal is to force clarity into your own head. Often you’ll see the real bug in the process.

07 · Tooling

Use What the Language Gives You

Languages evolve. A senior engineer writes modern idiomatic code — not the version of the language they learned in college. The modern forms are almost always shorter, safer, and more familiar to new team members.

Java
TypeScript
Python

Java modern forms worth adopting

  • record for immutable data carriers instead of verbose POJOs.
  • var for local variables with obvious types, to reduce noise.
  • Switch expressions with pattern matching — no more fallthrough bugs.
  • Text blocks (""") for multi-line strings instead of concatenation.
  • Optional at API boundaries; never Optional as a field.
  • Stream for transformations, but fall back to a simple loop when the pipeline becomes harder to read than the loop.
  • Sealed classes for closed type hierarchies — exhaustive switch checks at compile time.

TypeScript patterns to reach for

  • Discriminated unions over any or enum-with-payload patterns.
  • const by default; let only where reassignment is real.
  • Optional chaining (?.) and nullish coalescing (??) over defensive if chains.
  • Narrow types at the boundary (Zod, a parser); trust them inside.
  • Readonly arrays and objects where mutation isn’t intended.
  • satisfies for config objects where you want a widening check without losing literal types.

Python features that keep code modern

  • Type hints throughout — mypy or pyright in CI.
  • dataclass and TypedDict over dicts-of-anything.
  • Structural pattern matching (match/case) where branching on shape.
  • pathlib over string-based path munging.
  • f-strings for formatting; log with logging’s lazy %s form.
  • async/await for concurrent IO; comprehensions for transformations.
🔮

Why this matters: when the codebase uses modern idioms consistently, new hires ramp faster, senior devs from other teams contribute without friction, and the gap between your code and industry norms stays small. Drift is expensive.

08 · Diagnosis

Code Smells Catalog

A smell isn’t a bug. It’s a hint that something may be off — worth pausing over. Seasoned engineers develop a nose for these and address them while they’re small. The canonical catalogue is Fowler’s Refactoring; refactoring.guru and sourcemaking.com have excellent illustrated versions, free online.

Bloaters — things that grew too big

Long method

Scrolls past a screen. Usually doing more than one thing.
Why it matters: hard to read, hard to test, hard to change without breaking something else.
Move: Extract Method into named helpers — each helper’s name documents its chunk.

Large class

A class doing too much. A thousand-line class with 40 methods almost always wants to be two or three smaller classes.
Move: Extract Class; split by responsibility or by the subset of fields a group of methods uses.

Long parameter list

More than four or five parameters. Hard to call correctly; easy to get the order wrong.
Move: Introduce Parameter Object or Preserve Whole Object. Often the parameters have a natural grouping.

Primitive obsession

Raw strings and ints everywhere.
Why: String email and String firstName are the same type — the compiler won’t catch you mixing them up.
Move: Replace Primitive with Value Object — a UserId, Money, EmailAddress with validation on construction.

Data clumps

The same three fields keep appearing together (street, city, zip). They want to be one type.
Move: Extract Class; give the clump a name.

Object-orientation abusers

Switch statements on type

A switch that branches on the type or role of an object. Usually polymorphism is trying to get out.
Move: Replace Conditional with Polymorphism or Strategy.

Refused bequest

A subclass that doesn’t use (or actively contradicts) its parent’s methods. A Liskov violation waiting to bite.
Move: Replace Inheritance with Delegation; often the subclass shouldn’t inherit at all.

Temporary field

A field only used in certain operations; empty the rest of the time.
Move: Extract Class for the operation; put the field there, where it belongs.

Change preventers — make change harder than it should be

Shotgun surgery

A single conceptual change forces edits in 5–10 files.
Why: related knowledge is scattered.
Move: Move Method / Move Field to consolidate; aim for “one reason to change touches one place.”

Divergent change

A single class changes for many different reasons — the inverse of shotgun surgery. The class is doing too many jobs.
Move: Extract Class; each resulting class has one reason to change.

Parallel inheritance hierarchies

Every time you add a subclass in hierarchy A, you also have to add one in hierarchy B.
Move: Merge Hierarchies or replace one with composition.

Dispensables — things that add nothing

Duplication

The same three lines in four places. Real duplication — knowledge duplicated, not just similar shapes.
Move: Extract Method / Extract Class / Pull Up Method — after you’ve seen the third duplicate and the abstraction feels right.

Dead code

Unused method, unreachable branch, commented-out block.
Why: it costs reader attention and hides real behavior. Git remembers; your teammates don’t.
Move: delete it.

Speculative generality

Interfaces, base classes, and extension points for needs that haven’t arrived.
Move: collapse. When the need arrives you’ll design it better with real information — and the speculation will already be in the way.

Comments that apologise

“TODO: this is ugly.” “Hack — fix later.”
Why: apologies that don’t get fixed rot into folklore.
Move: either fix it now, or convert the apology into a tracked ticket (see tech debt tracking) with an owner and a date.

Couplers — things stuck together that shouldn’t be

Feature envy

A method that keeps reaching into another object’s fields. It wants to live on that other object.
Move: Move Method to where the data is. This is “Tell, Don’t Ask” applied in refactor form.

Inappropriate intimacy

Two classes know each other’s internals. Every change in one forces the other to change.
Move: Move Method / Move Field; or introduce a cleaner interface between them.

Message chains

a.getB().getC().getD().doThing(). Violates Law of Demeter; couples the caller to the whole chain.
Move: Hide Delegate — expose a single method on a that returns what the caller needs.

Middle man

A class whose methods all just delegate to another class.
Move: Remove Middle Man; let callers talk to the real class, or justify why the wrapper adds value.

📚

Deeper reading: refactoring.guru/refactoring/smells illustrates each smell with before/after code in multiple languages. martinfowler.com/bliki covers named concepts like AnemicDomainModel, TellDontAsk, and many more in short, clear entries. Both are free and worth an afternoon.

09 · Structure

Boundaries & Layering

A codebase that holds up over years has clear boundaries. Domain logic doesn’t reach into HTTP specifics; persistence doesn’t reach into UI state; each layer depends only on the layer below it.

The dependency rule

Draw an arrow from each layer to what it depends on. The arrows should point inward — toward the domain — never outward. The domain should know nothing about the HTTP framework, the database, or the UI. The adapters know about both sides; the domain knows only itself.

When you see a domain class importing javax.servlet or fastapi or @angular/core, that’s a boundary violation. Fix it before it spreads.

🪝

Signal of good layering: you could rewrite the HTTP framework, swap the database, or change the UI, and the business logic would barely change. You probably won’t do any of those — but the code that was written as if you might is almost always the cleaner code.

10 · Self-Check

The Six-Month Test

Before you mark a PR ready, ask: if I came back to this file in six months, with no memory of writing it, would I thank past-me or curse them?

Key Insight

The six-month test is a stand-in for “would a teammate thank me?” You’re the fairest reviewer of your own work because you can’t hide behind knowing what you meant. Six-month-you doesn’t remember what you meant either.

Questions for the test

  • Can I tell what this function does from its name alone?
  • Can I tell why this class exists from its module’s top comment or the surrounding code?
  • If this code breaks in production, would I know where to look?
  • If I need to change one thing here next quarter, is the blast radius small?
  • Is there any cleverness that a tired teammate would misread?
