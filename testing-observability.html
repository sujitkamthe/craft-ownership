Testing & Observability

Confidence in what you shipped, and visibility into how it’s doing. These two practices are what turn “it works on my machine” into “it works in production, and I can prove it.”

01 · Strategy

Test Strategy per Layer

The test pyramid isn’t a rule about counts — it’s a rule about where each test should live. Different kinds of bugs want different kinds of tests. Putting the test at the wrong layer is how teams end up with suites that are slow, flaky, and still let bugs through.

What each layer is for

  • Unit tests — for branching logic, math, parsing, calculation. Cheap, millisecond-fast, run on every save. If the bug lives in a pure function, this is its home.
  • Integration tests — for the wiring: your service talking to the database, the queue, the auth provider. Confirms the glue works. Slower, but catches the failures unit tests can’t see.
  • Contract tests — for “do the shapes match” across service boundaries. Much cheaper than full end-to-end.
  • End-to-end tests — for the few critical user journeys. Expensive and flaky — use sparingly, keep them to the journeys where a break would embarrass the team.

Why the pyramid matters in practice

🔍

Scenario: you want to verify that an invalid discount code shows an error. Three places could host the test — a unit test on the validator, an integration test on the API, or an end-to-end test that opens a browser, logs in, and clicks. The unit test runs in 2ms, never flakes, and pinpoints the bug. The end-to-end test runs in 45 seconds, fails twice a week for unrelated reasons, and when it fails you still don’t know why. Same bug. Vastly different test cost. Put the test where the bug actually lives.

Common pitfall

Inverted pyramid: thousands of slow end-to-end tests, few unit tests, flaky CI. The team loses faith in the suite; failures are re-run until green instead of investigated.

Better approach

Match each kind of risk to the cheapest layer that can catch it. Re-home tests that live at the wrong layer as you find them — it’s a legitimate ITC loop.

02 · Design

Writing Testable Code

Code that’s hard to test is usually code that’s hard to reason about. If adding a test feels painful, the test is showing you a design problem — listen to it.

Three moves that make code testable

  1. Separate pure logic from side effects. Put the decision in a pure function; let the caller do the IO. The decision is trivially testable; the IO can be tested once, separately.
  2. Inject dependencies at the boundary. Don’t reach for a global clock, a static database connection, or new’d collaborators inside a method. Receive them. Tests can pass fakes.
  3. Make time, randomness, and IO explicit. A function that calls LocalDateTime.now() internally is a nightmare to test. A function that takes a Clock argument is trivial.

Scenario

🧠

You have a TrialExpirationService that reads “now” from the system clock. To test what happens when a trial expires, your test has to freeze real time, or run against mocked static methods, or — the honest option — refactor the service to accept a Clock. The refactor takes 10 minutes and every test after it becomes easy.

Key Insight

If you find yourself reaching for heavy mocking frameworks to test ordinary code, pause. The test is trying to tell you the design is too coupled. Fix the design; the test will become simple.

03 · Practice

TDD — When It Helps

Test-driven development is not a universal rule. It’s a tool with a sharp point. Used where it fits, it’s the fastest way to write small, correct code. Used where it doesn’t fit, it feels like overhead.

Where TDD earns its keep

  • Pure logic with clear inputs and outputs — pricing, validation, state machines, parsers.
  • Bug fixes — write the failing test first; the fix must make it green.
  • API contracts — start from the interface a caller wants, then implement behind it.
  • Refactoring behind green — having a test to hold your hand through a structural change is worth the investment.

Where TDD fights you

  • Early exploration — you don’t yet know what the code should do. Sketch first, tests after.
  • UI visuals — a test can’t tell you whether the layout is right.
  • Spikes and prototypes — tests on throwaway code waste time.
🔮

The realistic compromise: rough out the shape in a sketch; once you know what the function should be, switch to TDD for the logic. You get the guided feel of TDD without paying for it on the parts of the problem where it doesn’t help.

04 · Quality

Integration Tests That Aren’t Brittle

An integration test fails for a real reason or for no reason at all. The first kind teaches you something; the second destroys trust in the whole suite. A good senior engineer refuses to ship flaky tests — including their own.

Rules that cut flakiness

  • Test behavior, not implementation. Assert on what the caller can observe, not on how the internals arrived there.
  • No shared mutable state between tests. Each test sets up its own fixtures and tears them down. No “test order matters.”
  • No sleeping for timing. Use event hooks or polling with a sensible timeout.
  • Control time, randomness, and IDs. Inject them. A flaky test that passes on Monday and fails on Tuesday is usually one of these three.
  • Zero tolerance for “re-run and move on.” If a test flaked, quarantine it, investigate, fix root cause. Every re-run-until-green normalises the noise.
Common pitfall

The test that asserts the exact SQL that was executed, or the exact sequence of internal calls. Any refactor breaks it, even when behavior is unchanged. The suite becomes an obstacle to change.

Better approach

Assert on the result — “after this call, user X has Y rows” or “the response matches this shape.” Refactors can happen freely; only real behavior changes fail the test.

💥

Why this matters: every team with a flaky suite eventually pays for it — in the bug that slipped through the noise, in the hours spent re-running CI, in engineers who stop trusting red builds. Brittle tests rot the team’s feedback loop.

05 · Production

Observability Baked In

Observability is not “ops’s job.” It’s part of shipping a feature. If your feature runs in production and you can’t tell whether it’s healthy without re-reading the code, you haven’t finished shipping it.

The three signals

Logs

Structured (JSON or key-value). Include correlation IDs. Enough context to answer “what happened to this request?” — no more. No PII, no secrets, no “entering method.”

Metrics

Counters and histograms for the outcomes the business or the operator cares about: requests per second, P95 latency, error rate per endpoint, queue depth, time-to-first-byte.

Traces

A single request’s path through services. Essential when a call crosses three services and takes too long — a trace tells you which hop is slow in seconds, not in grep.

The observability checklist before you call a feature done

  • There is a dashboard showing the health of this feature at a glance.
  • An alert fires on the signal a human would need to respond to — no more, no less.
  • A correlation ID flows from entry point to any downstream call.
  • Error logs include enough context to debug without a replay.
  • The on-call can answer “is this feature healthy right now?” without asking you.
📋

Scenario: at 2am, a page fires: the payment feature is failing for 8% of users. You’re the on-call’s friend. If you wired in clean logs with correlation IDs, clear metric names, and a trace on the critical path, they find the root cause in 15 minutes. If you didn’t, they’re still grepping at 4am. Observability is a gift to future-you and future-teammate.

Common pitfall

“I’ll add logs if something breaks.” Logs added after an incident are always partial — you can only log what you remember was involved. By then, you’ve already lost the first incident.

Better approach

Decide the observability plan during design, not during incident. Know what dashboard, what alerts, and what log fields you’ll need — before the feature ships.

06 · Safety

The Security Mindset

Security isn’t a specialist topic for a specialist team. It’s a set of habits every senior engineer carries into every feature. Most breaches are not exotic — they are the OWASP basics not being honoured.

Security habits that belong in every feature

  • Validate input at the boundary. Assume callers are hostile until proven otherwise. Use a parser (Zod, Pydantic, Jackson) rather than hand-rolled checks.
  • Authenticate, then authorise. Who are they, and are they allowed to do this specific thing? Never skip the second half.
  • Never trust the frontend. Any check you do in the browser, re-do on the server.
  • Parameterise queries. String-built SQL is a CVE waiting to happen. Use the parameterised form of your driver, always.
  • Escape on output. Context-appropriate: HTML, attribute, URL, JSON. Use the framework’s built-in escaping; don’t reinvent it.
  • Secrets live in a secret manager. Not in code, not in env files committed to git, not in Slack. And — don’t log them.
  • Keep dependencies patched. Dependabot or equivalent. A six-month-old known CVE is your fault, not the vendor’s.
  • Rate limit user-facing endpoints. You don’t want a simple script to take down your service.
🔒

Scenario: you’re adding an “admin deletes user” endpoint. The obvious thing is to check that the caller’s token says “admin.” The non-obvious thing: log who deleted whom and when, make the action undoable for 30 days, and rate limit it. If you ever face a rogue insider or a stolen admin token, those three details are what save you.

Key Insight

Security is a habit, not a phase. The engineer who remembers to parameterise the query at write-time doesn’t need a security audit to catch it at review-time. Build the habit once; it protects every feature after.

07 · Rollout

Feature Flags for Safe Rollout

A feature flag lets you ship the code without turning on the feature. That decoupling is the difference between “we can deploy any time” and “we need a two-hour maintenance window.”

Patterns that pay off

  • Dark launch. The new code path runs, its result is compared to the old path, but only the old path is served. You get real production data before your users see anything change.
  • Progressive rollout. 1%, 10%, 50%, 100%. At each step, watch the metrics. If the error rate or latency moves, you roll back to the last safe percentage.
  • Kill switch. The production setting that disables the feature instantly, without a deploy. Every non-trivial feature should have one.
  • Flag hygiene. Every new flag has a removal date. Old flags are as dangerous as bad dependencies — they add surface area and rot the code.
🛠️

Scenario: you’re swapping the checkout total calculation. The old one has edge cases you don’t fully understand; the new one is cleaner. You deploy dark: for 48 hours both run, but only the old number is shown to users, and you log divergences. You find two edge cases the old code handled accidentally-correctly. You fix the new code. You flip the flag. Zero customer impact. This is what real “deliver without bugs” looks like — it’s not brilliance at code review; it’s discipline at rollout.

Pick one this week:

  • Find a feature you shipped that has no dashboard. Build one. Note the metric that matters most and the signal you’d alert on.
  • On your next bug, write the failing test before you fix. Feel the difference in confidence when it goes green.
  • Pick one test in your suite that you’ve re-run to pass before. Fix it or quarantine it today — not next sprint.
