01 · Modelling

Data Modelling as a Design Decision

Your data model is the longest-lived thing you’ll ship. Code gets rewritten; schemas carry on for years with data nobody wants to lose. Treat the model as a first-class design decision, not something to rush past on the way to the “real” work.

Key Insight

Read queries come much more often than writes for most systems. If your model is easy to write and painful to read, you’ll feel it every day. Design for the questions the system needs to answer, not just the shape of the entities.

Questions to ask before you design a table

  • What queries will this serve? List them. Model around them.
  • What uniqueness, referential, and value constraints are real? Encode them — the database will catch bugs your code won’t.
  • What’s the expected row count in a year? Thousand, million, billion? The answer changes indexes, partitioning, and lifecycle.
  • What’s the access pattern? Bulk read, point lookup, range scan, full scan? Different patterns want different designs.
  • What will change? Columns you expect to evolve; columns you never want to evolve.

Normalise until it hurts, denormalise until it works

The rule of thumb is real: start normalised; introduce denormalisation deliberately when measurement shows the read cost justifies it. Premature denormalisation is how data gets inconsistent and stays that way.

Common pitfall

A JSONB column called metadata that becomes the dumping ground for every field nobody wanted to think about. Within a year, nobody knows what’s in there, queries over it are slow, and changes are scary.

Better approach

Reserve loose columns for genuinely variable data (user-defined fields). For known fields, name them, type them, constrain them. Your future self and your reports will thank you.

🔍

Indexes are design, not an afterthought. “We’ll add it when queries get slow” often means “we’ll add it during an outage.” At design time, list the queries, list the indexes that serve them, agree the plan.

02 · Craft

SQL Literacy for Engineers

Most senior backend engineers touch SQL regularly. Being comfortable reading a query plan, spotting an N+1, and choosing between a join and two queries is the difference between a service that scales and one that quietly brownouts at 2pm.

Read the query plan

Every database shows you how it plans to execute a query: EXPLAIN in Postgres and MySQL, EXPLAIN ANALYZE for the real numbers. You should be able to look at one and see the major shapes: sequential scan (reads the whole table) vs index scan vs index-only scan; nested loop vs hash join vs merge join; estimated rows vs actual rows.

Plan-reading signals that deserve attention

  • Sequential scan on a large table. Usually means a missing index or a query that can’t use one.
  • Estimated rows very different from actual. Stats are stale or the planner is wrong — maybe needs ANALYZE or a query rewrite.
  • Nested loop over a big outer. You probably want a hash or merge join; the planner may need more memory or better statistics.
  • Sort after limit. Often fixable with a matching index.

The N+1 problem, explained once

Bad — N+1 queries 
orders = db.query("SELECT * FROM orders WHERE user_id=?", uid)
for order in orders:
    order.items = db.query(
      "SELECT * FROM items WHERE order_id=?", order.id
    )
Good — one join, or two batched queries 
rows = db.query("""
  SELECT o.*, i.*
  FROM orders o
  LEFT JOIN items i ON i.order_id = o.id
  WHERE o.user_id = ?
""", uid)
# or: fetch orders, then items WHERE order_id IN (...)

N+1 is the single most common performance mistake in ORM-heavy codebases. Know the shape, spot it in PRs, log it in the reviews you give.

📈

Turn on query logging in your local and test environments. Seeing every SQL statement your code produces — and the count — is how you catch N+1 before a customer does.

03 · Safety

Migrations That Ship Safely

A bad migration can lock a table, fail mid-run, or leave prod in a state you can’t roll back from. Senior engineers treat migrations with the care they’d give a payment change.

The expand-contract pattern

How to change a schema without downtime

  1. Expand: add the new column, table, or relation. Nothing reads it yet.
  2. Backfill: populate the new shape from existing data. In batches, with pause points.
  3. Dual-write: code writes both old and new while readers still read old.
  4. Shift reads: readers switch to the new shape, feature-flagged. Old shape still consistent.
  5. Contract: once readers are all on the new shape and stable, stop writing the old, then drop it.

Feels tedious. Takes a few deploys. In exchange, you never have to schedule downtime and you can roll back at any step.

Common pitfall — the locking migration

ALTER TABLE large_table ADD COLUMN foo NOT NULL DEFAULT 'x'. On a big table in Postgres or MySQL, this can lock the table for minutes. The service goes down while you wait.

Better approach — split the migration

ADD COLUMN foo (nullable, no default) → backfill in batches → SET NOT NULL as a separate step once data is populated. No lock, no outage.

⚠️

Every migration should be rollback-compatible with the previous version of the app. If rolling back the code breaks because of a migration already applied, you’ve trapped yourself in the new state. Expand-contract is mostly about avoiding this.

04 · Interface

API Design — Hard to Misuse

A well-designed API guides callers toward correct use and makes incorrect use obvious. A poorly designed API invites the mistakes you’ll debug six months later.

Principles that compound

  • Model the domain, not the database. Your API is a contract about what the system does, not a thin veneer over tables.
  • Name endpoints around resources, actions around verbs. REST, RPC, or GraphQL — the caller should know what they’re doing from the name.
  • Make invalid states unrepresentable. If a field only applies when another is set, encode that in the type, not in “documentation please read.”
  • Fail loudly with useful errors. A 400 with “invalid request” teaches nothing. A 400 with which field, what was wrong, what’s allowed teaches.
  • Design for pagination from day one. Any endpoint that returns a list will one day return a lot. Cursor-based pagination beats offset for anything high-churn.
  • Make side effects obvious. A GET that mutates is a trap. A POST that pretends to be idempotent when it isn’t is worse.

Two small examples, one big difference

Easy to misuse 
POST /users/update
{ "id": 42, "email": "x@y.com",
  "admin": true }    // anyone can set?
// Errors: 500 "oops" on anything wrong
Guides the caller 
PATCH /users/42
{ "email": "x@y.com" }   // only allowed fields
// 403 if not permitted, 400 with per-field
// validation: {"email": "must be valid"}
📖

Before you expose an endpoint, write the three clients that will call it first — at least in your head. What’s awkward for them? What will they get wrong? The feedback often reshapes the API into something simpler and safer.

05 · Change

Backward Compatibility & Deprecation

Once an API has real callers, it’s no longer yours alone. Every change is a contract change. Senior engineers treat compatibility with the care it deserves — not as red tape, but as respect for the people who depend on you.

Compatibility-safe changes

  • Adding a new optional field.
  • Adding a new endpoint.
  • Relaxing validation that wasn’t strict enough.
  • Returning additional data in a response (if callers are not strictly schema-checking).

Breaking changes — require a versioning path

  • Removing or renaming a field.
  • Changing the type of an existing field.
  • Making an optional field required.
  • Tightening validation in a way existing callers violate.
  • Changing response shape or status code semantics.

The deprecation path

  1. Announce the change, with a date. Log deprecation warnings from the server.
  2. Build the new version alongside the old.
  3. Migrate known callers. Track them.
  4. Monitor usage of the old version until it drops to near zero.
  5. Remove. Communicate loudly before the removal date.
Common pitfall

Silent breaking change. A PR renames a field, ships, and five downstream teams see mysterious failures Monday morning. The explanation is in the git log — but none of them will read it.

Better approach

Flag breaking changes early in design review. Tag them in changelogs. Version them explicitly. A breaking change is a conversation, not a push.

06 · Reality

Distributed Systems Gotchas

Once your feature talks to another service over a network, the rules of single-machine programming stop applying. Networks fail partially. Calls time out. The “at most once” message is actually “at least once”. Senior engineers design for this, or pay for it later.

The short list of gotchas every senior should have internalised

  • Timeouts on every network call. No timeout = hang forever under partial failure. Default timeouts from your HTTP client are often too long.
  • Retries with exponential backoff and jitter. Naked retries during an outage amplify the outage. Jitter prevents synchronised stampedes.
  • Idempotency keys on writes. If a retry might arrive, make it safe. The caller passes a unique key; the server de-duplicates. Non-negotiable for payments, messages, and any action you don’t want to double.
  • At-least-once > exactly-once. Most message systems guarantee at-least-once. Design consumers to be idempotent; exactly-once is a fairy tale at the infrastructure level.
  • Circuit breakers for calls to services that can fail or slow down. Fail fast and recover deliberately beats queue-until-death.
  • Read-after-write isn’t guaranteed in most distributed stores. Don’t write then immediately read and expect the new value — unless you know the consistency model says so.
  • Clocks drift. Don’t compare timestamps across machines for ordering; use logical clocks or the database.
  • Queues can back up. A slow consumer becomes a huge consumer backlog. Monitor queue depth as a first-class signal.
🚨

Scenario: your service calls a payment provider. No timeout. The provider has a bad Friday afternoon. Your threads pile up waiting; your thread pool fills; healthy traffic stops being served. What started as their problem is now yours, for everyone. A two-second timeout and a circuit breaker would have contained it.

Key Insight

Distributed systems punish optimism. Every call that might fail, will; eventually. Build as if that’s true. Kleppmann’s Designing Data-Intensive Applications (see resources) is the best single book on why these gotchas matter; read it when you can.

07 · Last Resort

Caching — When, Where, and the Two Hard Things

Caching is often reached for too early, placed in the wrong layer, and sized by feeling. Done well, it removes real load. Done badly, it becomes its own source of bugs no dashboard shows.

Before you reach for a cache, ask:

  1. Have you measured? Is the database actually the bottleneck? Often it isn’t.
  2. Could the query be made faster? A proper index, a rewrite, or a materialised view is simpler than a cache.
  3. What is the cache invalidation rule? Time-based, event-based, version-based? If you can’t state it clearly, you’ll ship stale data to users.
  4. What does the system do when the cache is cold, full, slow, or unavailable? Cache failures should degrade gracefully, not cascade.
😁

“There are only two hard things in computer science: cache invalidation and naming things.” The joke is older than most teams and still correct. Take both seriously.

Patterns that work

  • Cache-aside for reads. Check cache; on miss, read from DB, populate cache, return. Simple, predictable.
  • TTL with stampede protection. Expiry + jitter + single-flight so you don’t get N simultaneous misses when the key expires.
  • Event-driven invalidation when you control the writes. The write publishes, cache subscribes, stale data is prevented.
  • Version in the key when you need immediate flip — change the version, the old key is abandoned.

Pick one this week:

  • Pick a slow query in your service. Run EXPLAIN ANALYZE. Understand the plan. Decide whether it’s a missing index, a rewrite, or actually fine.
  • Find a network call in your code with no timeout. Add one. Think about what the caller should see when the timeout fires.
  • Pick an internal API you’ve shipped. Read its error messages as if you were a new caller. Rewrite one to be more useful.
