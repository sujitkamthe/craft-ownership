Debugging & Reading Code

The skills seniors are quietly expected to have, and juniors are quietly expected to grow into. Debugging well and reading before changing are what separate engineers who fix things from engineers who keep breaking them.

01 · Method

Debugging Is a Scientific Method

A bug isn’t mysterious. It’s behavior produced by code, running on a machine, with inputs and state. When you treat debugging as guesswork, you get lucky sometimes. When you treat it as the scientific method — observe, hypothesise, test, repeat — you get fast reliably.

The debugging loop

  1. Observe. What exactly is happening? Not “it’s broken.” A specific input, producing a specific wrong output, in a specific environment.
  2. Hypothesise. What could cause this? List two or three candidates. Rank them by probability given what you know.
  3. Test one hypothesis cheaply. A log line, a breakpoint, a unit test, a query. Cheapest test first.
  4. Update your model. What did the test confirm or rule out? Eliminate branches.
  5. Repeat until you can point at the exact line. “I think it’s around here” is not done. Found is found.

The loop feels slow for the first few bugs. After ten bugs, it’s faster than guessing — and you actually know why the bug was there, which means you can prevent the class, not just the instance.

Key Insight

Most long-running bug hunts are really short hunts repeated — the debugger changed something, forgot what, changed something else, lost the thread. Keep a cheap log while you debug: what you tried, what you saw. Five lines in a scratch file saves hours of “wait, did I test this already?”

This takes years to feel natural. Early in your career, you’ll catch yourself guessing; name it when you do and switch back to the loop. The habit forms by doing.

02 · Discipline

Reproduce Before You Fix

A bug you can’t reproduce is a bug you can’t be sure you fixed. Reproducing is the single most valuable step and the one most often skipped.

The minimal repro ladder

  • Reproduce in the original environment. The full app, the production-like data — confirm the behavior exists.
  • Shrink the input. What’s the smallest input that still triggers it? One user? One row? One request?
  • Shrink the code path. Can you reproduce it in a unit test or a 20-line script? If yes, you’ve isolated it.
  • Lock the repro as a test. Before you fix anything, write the failing test. The fix is done when the test goes green.
🔍

Scenario: “Customers see wrong total at checkout” — vague and scary. You open support tickets: three customers, one thing in common — all used a 20% discount code on an item that was already discounted. You try it: yes, reproduces. You shrink: same bug with a 10% code, any item, second-discount flow. You write a unit test that fails. Now you have a named bug, not a mystery. The fix takes 15 minutes; the confidence it’s actually fixed is absolute.

Common pitfall

“It’s probably this.” You change the code, can’t reproduce the bug (because you never could), ship the fix, cross your fingers. Three weeks later the customer is back.

Better approach

If you can’t reproduce, spend time getting a repro before spending time on the fix. Often the repro attempt is how you find the bug in the first place.

03 · Technique

Bisection — Halving the Haystack

When you know the bug wasn’t there last week and is there today, bisection finds the commit that introduced it in log(N) steps, not N. It’s one of the highest-leverage techniques you can learn.

git bisect in 60 seconds

git bisect start
git bisect bad                 # current commit is broken
git bisect good v1.4.0         # this tag was good
# git checks out a midpoint; you test; then:
git bisect good                # or: git bisect bad
# repeat until git prints the offending commit
git bisect reset

If you have an automated test that reproduces the bug, git bisect run ./repro.sh does the whole thing for you in one command.

Bisection as a habit, not just a command

The habit extends beyond git. When something is broken: what’s the smallest thing I can change to learn whether my model is right? Remove half the inputs. Comment out half the code. Query half the users. Eliminate halves until the problem narrows to a point. Almost every hard debug collapses into easy steps under bisection.

✂

When a library upgrade broke something, git bisect across the dependency’s changelog is often faster than reading the release notes.

04 · Reading

Read the Stack Trace, Don’t Guess

Every error message and stack trace is a free tip from the runtime about where the bug lives. Most time spent “debugging” is really time spent not reading what was already written for you.

How to read a stack trace

  1. Read the message. All of it. Twice. The type, the text, any context the framework added.
  2. Find the top frame that’s in your code. Framework frames are usually noise; your code is usually where you can act.
  3. Understand the call path. Who called the failing function, with what? Sometimes the bug is three frames up.
  4. Note line numbers against the current commit. Traces from old deploys point at old line numbers — confirm which version produced the error.
  5. Look for patterns across multiple occurrences. One trace is a data point; five similar ones are a signal.
🔎

Scenario: a teammate pings you: “got this stack trace, any ideas?” You open it. The message is a NullPointerException at line 42 of OrderService. Twenty seconds of reading: line 42 is order.getPayment().getAmount(). Either order or getPayment() returned null. Ask: “can order exist without a payment?” Yes. There’s the bug. Two minutes of reading saved an afternoon of “let me try adding some logs.”

05 · Before Changing

Read the Code Before You Change It

Blind changes cause regressions. Reading the code you’re about to edit — the function, its callers, its tests, its recent history — takes minutes and often changes what you were going to write.

A pre-change reading routine

  1. Read the function end-to-end. Not just the lines you intend to touch. What does it promise?
  2. Read its tests. What behavior is already guaranteed? What invariants might your change break?
  3. Read the callers. How is this used? Are there assumptions you’d be breaking?
  4. Read the recent git log for the file. Why was it last changed? By whom? Any related commit messages that explain the current shape?
  5. Ask the author or recent committer if the change is significant. Five-minute conversation beats two days of undoing a subtle regression.

Key Insight

The worst PR reviews I’ve seen aren’t wrong code — they’re correct changes made in ignorance of invariants the code was quietly upholding. Reading the surrounding 50 lines before touching the 5 you came for prevents most of those.

Common pitfall

Jumping straight to the line number the IDE took you to, typing your change, submitting. The change works in isolation; it breaks something the surrounding code depended on.

Better approach

Before you type, scroll up and down. Read the function. Run the tests. Skim git blame. Often you’ll spot a better place to make the change, or realise the change isn’t needed at all.

06 · New Codebase

Ramping Up on a New Codebase

Joining a new project is a repeatable skill. Senior engineers have a routine: 30 days to go from stranger to confident contributor, done the same way every time.

The 30-day ramp-up

  1. Day 1–3: run it. Clone, build, start the service, run the tests. If any step fails, improve the README or ask — don’t suffer in silence. Your fresh eyes will spot friction veterans no longer see.
  2. Day 4–7: read the top-level structure. The main module, the key entry points, the domain model. Draw the module graph on a napkin if it helps.
  3. Day 8–14: trace one user journey end-to-end. Sign-in, or a core feature, or a cron job. Follow the request from HTTP to DB to response. Note what surprised you.
  4. Day 15–21: small PR. A typo fix, a tiny bug, a missing test. Get through the review process. Learn the team’s rhythm.
  5. Day 22–30: own something small. A story, a refactor, a bug class. Drive it from request to deploy. You’re ramped.
📚

Scenario: you join a new team. On day 2, make dev fails because of a missing env var not in the README. The old-timers know about it and shrug. You spend 20 minutes, document it, open a PR to the README. Everyone who joins after you is saved those 20 minutes. Day 2 contribution, lasting.

Ask questions that earn you respect

Good ramp-up questions are specific, show you’ve done the homework, and respect the other person’s time:

  • “I see OrderRouter dispatches to three handlers. Is there a reason RefundHandler isn’t one of them?”
  • “The README says to run make seed, but that target doesn’t exist. Has it been renamed?”
  • “Can you walk me through how a payment flow reaches the gateway? I’ve traced the happy path but want to understand the retry logic.”

Versus: “How does this work?” — which asks someone else to do your first three hours of reading.

07 · Live

Debugging in Production

Production is where the real bugs live. The skills differ from local debugging: you can’t pause execution, you can’t change state, you have to reason from traces of what already happened. Good observability (baked in during build) makes this tractable; poor observability makes it impossible.

The production investigation order

  1. Check the dashboard first. Is there a graph that already shows the story? Error rate, latency, throughput — which one moved and when?
  2. Find a single failing request. One correlation ID, one user, one timestamp. One concrete example beats ten aggregated reports.
  3. Trace it across services. Which hop failed? Which call was slow? Distributed traces exist for exactly this.
  4. Read the relevant logs for that correlation ID. Structured logs make this one query; unstructured logs make it an archaeology expedition.
  5. Form a hypothesis; verify with another example. Did a second failing request show the same pattern? Third? Confidence grows.
🚨

Under load, mitigate first, diagnose second. If users are affected: kill-switch, shift traffic, roll back. Then investigate calmly from the logs. Debugging heroically in the middle of an incident is how incidents get longer.

08 · Measure

Profiling When It Matters

Engineering intuition about performance is usually wrong. The part you think is slow isn’t, and the part that is never occurred to you. Profile before you optimise, always.

When to reach for a profiler

  • A latency target is being missed and the code is non-trivial.
  • A service is spending money in ways that feel high for the workload.
  • A feature works but is “kind of slow” and users are starting to notice.
  • You’re about to make a “surely this will be faster” change — measure first so you can prove it.

Flame graphs — what to look for

A flame graph shows which functions burned CPU. The widest boxes are where the time went. You look for: one surprisingly wide frame (the hot spot), unexpectedly tall stacks (deep call paths), or broad flat areas where many calls each took a little (often database or network).

📈

Scenario: a checkout endpoint P95 has climbed from 200ms to 1.2s. The team suspects the new loyalty-points code. You take a flame graph: turns out the loyalty code is 40ms; the actual hot spot is a JSON serialiser pulling in fields it never uses. A two-line fix cuts P95 to 240ms. You ship. Nobody would have guessed.

Common pitfall

Optimising the code you suspect. Often you spend a day on a 2% improvement while a 40% improvement sits in code you never looked at.

Better approach

Spend 20 minutes measuring. Optimise the biggest contributor. Re-measure. Stop when you’ve hit the target; don’t polish for its own sake.

09 · Honesty

“I Don’t Know” Is a Senior Skill

One of the surest signals of a truly senior engineer: they say “I don’t know” quickly, without embarrassment, and then go find out. Confident pretending is a junior trap that some engineers carry for a decade before noticing.

Key Insight

Your teammates can usually tell when you’re fogging. They don’t call it out because it’s awkward. Your credibility slowly leaks anyway. Saying “I’m not sure — I’ll dig in and come back” costs nothing in the short run and pays compound interest in trust.

A template that works

  • “I don’t know — I’d guess X, but let me verify.” Honest, useful, ends with a commitment.
  • “I’ve seen this kind of thing before, but not this exact situation. My lean is X.” Calibrated confidence.
  • “Nobody I know has a confident answer on this. Let’s run an experiment.” Turns the unknown into a plan.
🎯

Close the loop. If you said “I’ll find out” on Monday, come back Tuesday with an answer. The credibility comes from the follow-through, not from the original admission.

Pick one this week:

  • On the next bug you’re assigned, write the failing test before you fix anything. Notice the difference in your confidence when the test goes green.
  • Try git bisect on a real regression this week. Learn the command before you need it in anger.
  • Before your next meaningful PR, spend 15 minutes reading around the code you’re changing — callers, tests, git log. Notice what changes in your approach.
